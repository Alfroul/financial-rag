# Failure Analysis — RAG 系统 Bad Case 分析

> 记录系统回答不佳的真实 case，分析根因，追踪改进效果。
> 最后更新：2026-06-07

## 分析维度

| 维度 | 说明 |
|------|------|
| 检索失败 | 相关文档未被召回 |
| 排序失败 | 相关文档被召回但排名靠后 |
| 生成幻觉 | LLM 编造了参考资料中没有的信息 |
| 上下文不足 | 检索到的文档不足以回答问题 |
| 领域知识缺失 | 文档库中完全没有相关信息 |

---

## Case 1: 混合检索 Faithfulness 下降（hybrid 0.66 vs vector 0.80）

**现象：** Round 9 benchmark 中，hybrid 检索的 Faithfulness（0.66）显著低于纯 vector（0.80）。

**根因分析：**
- BM25 引入的文档可能包含大量关键词匹配但语义不相关的片段
- RRF 融合时，低质量的 BM25 结果被提升到 top-k，挤掉了高质量的向量结果
- LLM 在更多低相关度上下文中更容易产生幻觉或忽略关键信息

**修复措施：**
1. 引入 BGE-reranker 精排，hybrid + Reranker 后 Faithfulness 回升到 0.76（+15.3%）
2. 后续可调整 RRF k 参数或 BM25 检索阈值，过滤低分文档

**效果验证：**
| 配置 | Faithfulness | 变化 |
|------|-------------|------|
| vector | 0.801 | baseline |
| hybrid | 0.661 | -17.5% |
| hybrid + Reranker | 0.762 | +15.3% |

## Case 2: 数值查询精度不足（Round 8-9 共性问题）

**现象：** 对于 "2024年A股IPO融资总额" 等精确数值查询，系统回答与参考答案有偏差。

**典型问题：**
- Q: "中国2024年GDP增速目标是多少？" → A: "约为5%"（正确但缺乏权威引用）
- Q: "2024年A股市场IPO融资总额约为多少？" → A: 数值偏差大（检索文档无最新数据）

**根因分析：**
1. **文档时效性**：知识库中的文档可能不包含 2024 年最新数据
2. **数值精度丢失**：分块（chunk_size=512）可能将数据与上下文分离
3. **LLM 舍入倾向**：MiMo/Qwen 模型倾向于给出近似值而非精确引用

**改进方向：**
- 增加文档更新频率，确保数据时效性
- 对数值类查询启用 self-correction，通过外部验证核实数值
- 调整 chunk 策略：数值密集段落使用更小的 chunk_size

## Case 3: 对比分析类查询上下文不足

**现象：** GraphRAG 路径的 Faithfulness（0.69）低于预期，对比类问题如 "沪深300和中证500的区别" 效果不佳。

**根因分析：**
1. **图谱覆盖不足**：知识图谱中实体关系有限，两家公司/指数之间的关联三元组少
2. **实体匹配失败**：用户查询 "沪深300" 但图谱中存储为 "沪深300指数"，匹配失败
3. **对比上下文缺失**：检索文档可能只包含单一实体的信息，缺少横向对比的数据

**改进方向：**
- 扩充三元组提取的规则，增加 "对比"、"关联" 类关系
- EntityMatcher 增加别名/缩写映射表
- GraphRouter 检测到对比意图时，主动对两个实体分别查询再合并

## Case 4: Agent 多步推理中的工具选择错误

**现象：** Agent benchmark 中，部分任务（如 CAGR 计算）需要 3 步，但 Agent 可能在第 1 步选择了错误工具。

**典型失败模式：**
- Task: "计算贵州茅台近3年的营收复合增长率" → Agent 选择 `calculator` 而非先 `financial_search` 获取数据
- Task: "央行降息对银行股的影响" → Agent 缺少 `knowledge_graph` 调用，无法获取行业传导链

**根因分析：**
1. **Few-shot 覆盖不足**：ReAct prompt 中的 few-shot 示例未覆盖所有工具组合
2. **LLM 推理局限**：MiMo-V2-Pro 在复杂多步规划上能力有限，不如 Claude/GPT-4
3. **工具描述粒度**：`calculator` 的描述未强调 "需要先检索数据"

**改进方向：**
- 增加 more few-shot examples 覆盖 edge cases
- 在 system prompt 中加入 "先检索，后计算" 的规则
- Task Classifier 在循环前检测是否需要先检索数据

## Case 5: 自我修正的误报问题

**现象：** Self-Correction 启用后，部分正确答案被标记为 "有疑点"（confidence < 0.5），导致不必要的重试。

**根因分析：**
1. **Claim NLI 严格匹配**：NLI 验证要求断言与上下文精确匹配，但 LLM 的合理推断被标记为不支持的断言
2. **外部验证 API 限制**：verifier 模型（MiMo-V2.5-Pro）对金融领域事实判断不够准确
3. **Rule Checker 误报**：规则 "不应包含具体数值预测" 可能将合理的趋势分析也标记出来

**改进方向：**
- 调整 NLI 阈值，允许合理的推断通过
- 增加金融领域规则白名单（如允许引用历史数据做趋势分析）
- 优化 Claim 粒度：将长断言拆分为更小的原子事实

## 总结

当前系统的主要瓶颈：
1. **检索质量**是最大的影响因素 — Reranker 带来 15.3% 的 Faithfulness 提升
2. **文档时效性**限制了数值查询的准确性 — 需要持续更新知识库
3. **图谱覆盖**还不够密集 — 对比类查询需要更丰富的实体关系
4. **Agent 规划能力**受限于 LLM 本身 — 可通过 prompt engineering 弥补

这些 bad case 的价值在于：它们揭示了 RAG 系统的性能瓶颈不在单一组件，而在**端到端链路的协同**。每个组件都有优化空间，但最大的收益来自最薄弱环节的改进。
